| Role | Size | Weight | Line Height | Font |

## Visual Hierarchy / Focal Points

Each screen state must have one declared focal point — the first element that draws the eye and orients the organizer.

| Screen State | Focal Point | Rationale |
|---|---|---|
| Dashboard — no competitions exist | "Create Competition" CTA button (centered, primary bronze) | Empty state: only action available; everything else is muted copy below it |
| Dashboard — competitions exist, none selected | Competition selector dropdown (top of page, full-width trigger) | The selector is the gate to all other content; nothing else is interactive until it is used |
| Dashboard — competition selected | Analytics bar stat numbers (28px bold, first visible content below selector) | Numbers answer "how many?" immediately; table below provides detail |
| Timetable view | Time column + first platform column pair (left-anchored) | Organizers scan the schedule top-to-bottom; the time column anchors the eye; the first platform column is where reading begins |
| Timetable view — conflicts present | Conflict panel (positioned above grid, expanded by default, red/amber text contrast) | Conflict panel renders before the grid in DOM order; high-contrast text color ensures it pulls attention before the grid |

## Interaction Contracts

### Competition Selector (DASH-01)
- Renders as: shadcn `Select` trigger showing selected competition name + status badge, or "Select a competition" placeholder
- On open: dropdown items show competition name (bold 14px), date (12px muted), status badge
- Items sorted: most recent first (matches server query `desc(competitions.createdAt)`)
- On select: URL updates to `?compId={id}` via `router.push` (no page reload, `useSearchParams` reads it)
- Initial load: if `?compId` in URL, auto-select that competition (read in server component, pass as prop)
- "New Competition" button: positioned right of selector, links to `/organizerdb/create`

### Analytics Bar (DASH-02)
- Always visible when a competition is selected
- Stat cards in a horizontal `flex-wrap` row with `gap-4`
- Conditional cards: "Spots Remaining" only if `max_registrants` is set; "Deadline" only if `registration_deadline` is set
- Deadline countdown format: "3 days" / "2 hours" / "Expired" (never raw datetime)

### Registrations Table (DASH-03)
- Client component; receives full registrants array as prop from server
- Sort state: `useState` with `{ column: string, direction: 'asc' | 'desc' }`
- Filter state: `useState` with `{ event: string | null, gender: string | null }`
- Column sort: click header toggles asc/desc on that column; clicking a different column resets to asc
- Event filter: pill buttons (All / LC / Jerk / Snatch) — filters to registrants who have that event
- Gender filter: pill buttons (All / M / F)
- Events column: renders event badges as pills (e.g. `Badge` with "LC", "Jerk", "Snatch")
- Remove action: trash icon button (`Lucide Trash2`) opens `RemoveRegistrantDialog`

### Remove Registrant (DASH-04)
- Dialog trigger: `Trash2` icon in table Actions column (tooltip: "Remove registrant")
- Dialog content: "Remove [Full Name]?" heading + "This will delete their registration and all serial numbers ([SER-0001, SER-0002]). This cannot be undone." body
- Buttons: Cancel (left, secondary style) | Remove Registrant (right, `bg-red-600 hover:bg-red-700 text-white`)
- On confirm: server action call, optimistic row removal, show toast/inline success
- On error: show inline error within dialog, keep dialog open

### CSV Import (DASH-05)
- Trigger: "Import CSV" button in dashboard action bar
- Modal sections:
  1. Drop zone: dashed border, "Drop CSV here or click to select" (12px muted), accepts `.csv` only
  2. Preview table: appears after file selected; columns match QUEUE_SPEC CSV format
  3. Row validation: valid rows get `bg-green-950/30` row tint; invalid rows get `bg-red-950/30` with error message in a rightmost "Error" column
  4. Summary bar: "13 valid, 2 errors" in 14px; errors list if any
  5. Action buttons: "Import 13 valid rows" (primary, disabled if 0 valid) | Cancel
- CSV parsing: client-side FileReader — no server call until "Import" is clicked

### Generate Queue Modal (DASH-06)
- Trigger: "Generate Queue" button in dashboard action bar
- Button disabled state when 0 registrants: cursor-not-allowed + `opacity-50` + tooltip "Need at least 1 registrant to generate queue"
- Modal content: start time input (`type="time"`, default "09:00"), summary ("N registrants across P platforms"), estimated finish time (calculated from scheduler or `startTime + blocks * 10min`)
- Confirm: redirects to `/organizerdb/queue?compId={id}&startTime={minutesSinceMidnight}`
- Cancel: closes modal, no navigation

### QR Code Modal (D-05, D-06)
- Trigger: "QR Code" button beside "Copy Registration Link"
- Modal content: QR code rendered as canvas or `<img>` encoding `{origin}/registration/{compId}`
- "Download QR as PNG" button: triggers canvas `toDataURL()` download
- Close: X button or click-outside

### Timetable Grid (SCHED-07)
- Server component at `/organizerdb/queue`
- Reads `compId` + `startTime` from URL search params
- Calls `schedule()` with entries from DB, passes result to client components
- Grid columns: Time | Block # | Platform 1 | Platform 2 | ... | Platform N
- Time column: formatted as "09:00", "09:10", etc. based on `startTime` + block index * 10min
- Empty platform slots: empty `<td>` (no filler text)
- Row event tint: applied via className on `<tr>` based on majority event in block (screen only)
- "Back to Dashboard" link: `←` icon + "Back to Dashboard" text, links to `/organizerdb?compId={id}`

### Conflict Panel (SCHED-08)
- Positioned between timetable header and grid
- Default state: expanded (no collapse animation needed, simple `<details>` or controlled `useState`)
- Expand/collapse toggle: "▼ 3 conflicts" / "► 3 conflicts" chevron button
- REST conflict lines: `text-red-400` — "REST: [Athlete Name] — blocks [N] and [N] (gap: N blocks)"
- COACH conflict lines: `text-amber-400` — "COACH: [Coach Name] coaching [Athlete Name] in block [N]"
- Zero conflicts state: `text-green-400` — "No conflicts"

### Print Layout (SCHED-09)
- `@media print` CSS in timetable page:
  - Hide: nav, "Back to Dashboard" link, "Generate Queue" button, ConflictPanel collapse toggle
  - Show conflicts inline as text above grid (no color — only bold "REST:" / "COACH:" prefixes)
  - Remove event tint backgrounds (`print-color-adjust: exact` NOT used — rely on B&W)
  - Page orientation: `@page { size: A4 landscape; margin: 10mm; }`
  - Print header: competition name (bold 16px), date, "Generated: {timestamp}" on one line
  - Table width: 100% with `table-layout: fixed` for even column distribution
  - Font size in print: 10px for cell content to fit landscape

## Phase-Specific Notes

### Desktop-First Requirement
REQUIREMENTS.md Out-of-Scope explicitly states: "Mobile-specific layouts for organizer pages — R2: desktop/PC browser first." All layout decisions use desktop constraints. The analytics bar wraps on narrow viewports (`flex-wrap`) as a graceful degradation only — it is not a mobile breakpoint target.

### No Auth Gate
CONTEXT.md (From Prior Phases): "No auth gate on organizer routes for v2.0 — code structured for Clerk retrofit." No `auth()` calls, no redirect-on-unauthenticated. The layout at `app/organizerdb/layout.tsx` must not be modified.

### Timetable is a Server Component
The timetable page at `app/organizerdb/queue/page.tsx` is a server component. The scheduler `schedule()` call happens server-side. The `TimetableGrid` and `ConflictPanel` receive pre-computed data as props. Client interactivity (collapse panel) may be a nested client component wrapper.

### db.batch() for All Mutations
All remove and bulk-import server actions use `db.batch()`. The executor must not introduce `db.transaction()`.

### Weight Class Display
`getWeightClass(gender, bodyWeightKg)` called at render time in timetable cell. Never stored or fetched from DB.
