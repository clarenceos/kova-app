<objective>
Fix AudioContext user gesture chain broken by async/await in the recording page.

Problem: On iOS Safari and some Android browsers, AudioContext requires creation and `.resume()` within a synchronous call stack from a user gesture (tap/click). Currently `handleStart()` immediately awaits `startSession()` which does async work (getUserMedia, video.play, font loading) BEFORE creating AudioContext on line 485. By the time AudioContext is constructed, the browser no longer considers it a user gesture context, so it starts in `suspended` state and beeps never play.

Additionally, `playBeep` and `playStartTone` each redundantly create a new AudioContext if one is missing and call `await ctx.resume()` -- this is both unnecessary (context will already exist and be running) and harmful (the async resume re-suspends the gesture chain for oscillator scheduling).

Fix: Create and resume AudioContext synchronously in `handleStart` before any await. Remove redundant creation/resume from `startSession`, `playBeep`, and `playStartTone`.

Output: Single modified file with AudioContext always created in user gesture context.
</objective>

**Change 1 — `handleStart()` (currently line 641):**
Add AudioContext creation and resume BEFORE the `await startSession(...)` call, synchronously in the click handler:

```typescript
async function handleStart() {
  const weightNum = parseFloat(weight)
  if (isNaN(weightNum) || weightNum < 4 || weightNum > 100) return
  if (countdown < 5) {
    setCountdownError('Minimum countdown is 5 seconds')
    return
  }

  // Create AudioContext synchronously in the user gesture call stack
  // so the browser allows audio playback. MUST happen before any await.
  audioCtxRef.current = new AudioContext()
  void audioCtxRef.current.resume()

  const isFront = cameras.find(c => c.deviceId === selectedCamera)?.label === 'Front Camera'
  await startSession(weightNum, countdown, beep, autoStopEnabled, selectedCamera, isFront)
}
```

Key: `void audioCtxRef.current.resume()` is fire-and-forget — the resume promise is not awaited so the gesture chain stays synchronous. The `void` operator makes the intentional fire-and-forget explicit and avoids lint warnings about floating promises.

**Change 2 — `startSession()` (currently lines 485-486):**
Remove the AudioContext creation line (`audioCtxRef.current = new AudioContext()`). Keep the destination creation but use the already-existing context:



This is safe because `handleStart` is the only caller of `startSession` and always creates `audioCtxRef.current` before calling it.

```typescript
const playBeep = useCallback(() => {
  const ctx = audioCtxRef.current

  const oscillator = ctx.createOscillator()

  oscillator.stop(ctx.currentTime + 0.3)
}, [])
```
