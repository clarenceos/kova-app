  <files>app/organizerdb/_components/QRCodeModal.tsx</files>
  <action>
Replace the canvas-based QR rendering with a data URL approach that is immune to the Dialog lazy-mount timing issue.

In QRCodeModal.tsx:
1. Remove the `canvasRef` (useRef for canvas). Replace with a `const [dataUrl, setDataUrl] = useState<string | null>(null)` state.
2. In the useEffect (deps: [open, compId]), when `open` is true:
   - Build the URL: `${window.location.origin}/registration/${compId}`
   - Call `QRCode.toDataURL(url, { width: 256, margin: 2, color: { dark: '#EDE9E2', light: '#0D0D0D' } })` which returns a Promise<string>
   - Set the resulting data URL string into state: `setDataUrl(result)`
   - When `open` is false, reset: `setDataUrl(null)`
3. Replace `<canvas ref={canvasRef} className="rounded-lg" />` with:
   ```tsx
   {dataUrl ? (
     <img src={dataUrl} alt="Registration QR Code" className="rounded-lg" width={256} height={256} />
   ) : (
     <div className="h-[256px] w-[256px] animate-pulse rounded-lg bg-charcoal" />
   )}
   ```
4. Update `handleDownload`: instead of reading from canvas, use the `dataUrl` state directly. Create an `<a>` element with `href=dataUrl` and `download` attribute, click it. Guard: if `!dataUrl` return early.

Why toDataURL over toCanvas: The shadcn Dialog mounts DialogContent lazily. When the effect fires on `open=true`, the canvas ref is still null because the DOM has not mounted yet. toDataURL is purely in-memory (no DOM element required), so it works regardless of mount timing.
