**Bug 1 — Multi-discipline registration fails:**

Investigate the RegistrationForm.tsx `handleSubmit` function. The form correctly builds `eventPayload` as an array of checked events and passes it to `registerAthlete`. The server action in `lib/actions/registrations.ts` loops over `input.events` and generates serials, then does `db.batch()` with one registrant insert + N entry inserts. The batch call uses spread: `...input.events.map((evt, i) => db.insert(...))`.

The likely cause is that `db.batch()` requires a tuple type, and the dynamic spread may fail TypeScript or runtime when the array has >1 entry. Check the actual error by testing with 2+ events. The existing code at line 99 does:
```typescript
await db.batch([
  db.insert(registrants).values({...}),
  ...input.events.map((evt, i) => db.insert(registrationEntries).values({...}))
])
```

The `db.batch()` call was cast via `unknown` in the bulk import (per decision Phase 10-01), but `registerAthlete` does NOT do this cast. Add the same cast pattern:
```typescript
const statements = [
  db.insert(registrants).values({...}),
  ...input.events.map((evt, i) =>
    db.insert(registrationEntries).values({...})
  ),
] as unknown as [typeof statements[number], ...typeof statements[number][]]
await db.batch(statements)
```

If the issue is NOT the batch type (test first), investigate whether the form is sending duplicate or malformed event data when multiple events are checked.

**Root cause:** The `--input` CSS variable is `#1A1A1A` — nearly the same as the dark `charcoal` background (`#1A1A1A` IS charcoal). The shadcn RadioGroupItem and Checkbox both use `border-input` for their unchecked border, making them invisible.

**Fix approach — update the `--input` CSS variable** to a visible border color for form controls. Change `--input` from `#1A1A1A` to `rgba(123, 140, 151, 0.3)` (raw-steel at 30% opacity) in BOTH the `:root` and `.dark` blocks in `app/globals.css`. This uses the same raw-steel color palette as `--border` but slightly more opaque so form controls are clearly visible.

Before changing, verify this won't break other elements using `border-input` — search the codebase for `border-input` usage. The shadcn Select trigger, Input, and other form elements also use `border-input`, but they already have explicit border overrides via the `inputClass` pattern (`border-raw-steel/30`), so the change should only affect the primitives (RadioGroupItem, Checkbox) that rely on the default.

**Alternative if `--input` change is too broad:** Instead of changing the CSS variable, override the border color directly in `RadioGroupItem` and `Checkbox` components by replacing `border-input` with `border-raw-steel/30` in both components. This is more surgical and avoids side effects.

Choose the surgical approach (replace `border-input` with `border-raw-steel/30`) to minimize risk:

In `components/ui/radio-group.tsx` line 29, in the className string, replace `border border-input` with `border border-raw-steel/30`. Keep all other classes unchanged. The `data-checked:border-primary` still overrides when selected.

In `components/ui/checkbox.tsx` line 17, in the className string, replace `border border-input` with `border border-raw-steel/30`. Keep all other classes unchanged.

Also replace `dark:bg-input/30` with `dark:bg-transparent` in both components, since `bg-input/30` on the `#1A1A1A` input color just adds more darkness. Transparent is cleaner.

This ensures:
- Unchecked state: visible `raw-steel/30` ring/outline (matches the existing inputClass convention)
- Checked state: `border-primary` (patina-bronze) kicks in via `data-checked:border-primary` — no change needed
- The ghost dot/ring is visible against charcoal, card, and background surfaces
